# GenCall v2.2.9 — Diagnosable SIPp launch failures

**A SIPp instance that fails at startup now tells you _why_.**

Until now, a failed launch reported only:

```
SIPp exited during startup (code 1); see SIPp -trace_err file
```

…but `stderr` was sent to `/dev/null`, `-trace_err` is off by default, and the
engine deletes that file on cleanup. So a failing launch was effectively a
black box.

## Why this mattered (a real incident)

The diurnal traffic shaper relaunches a campaign's UAC every hour to step its
rate along the day/night curve. When a **worker process outlived a code deploy**
that replaced its working directory, every relaunched SIPp inherited the
now‑deleted CWD and couldn't create its (relative‑path) `-trace_logs` file —
exiting `code 1`.

The shaper caught the failure, logged a one‑line warning, and **kept the old UAC
at the night‑floor rate** — about **1,400 times a day, for days**. The result: a
campaign sized for 200k minutes/day quietly delivered ~13k/day (~7% of target),
with no error text anywhere to explain the freeze. Diagnosing it took hours of
packet captures and live reproduction.

With this release, the same failure is a two‑second read.

## What changed

- **SIPp `stderr` is captured to a small per‑instance file** under the run
  directory (never a PIPE — an unread pipe can deadlock a busy run on a full
  kernel buffer).
- **Startup failures surface the real cause** in `error_message` and the log,
  e.g. `SIPp exited during startup (code 1): Unable to create 'loop_uac_…_logs.log'`.
- The capture file is cleaned up alongside the other per‑instance artifacts, so
  it can't leak or fill the disk.

The CWD hardening that prevents the failure (running SIPp from the writable
stats dir) already shipped earlier; this release makes the *next* such failure
self‑explanatory instead of silent.

## Upgrade notes

- No config or API changes. Drop‑in.
- **Always restart the worker after a code deploy** (`systemctl restart
  gencall-worker`). A long‑lived worker that outlives a reinstall of its install
  directory is the root condition this incident exposed — a restart re‑binds the
  current working directory.
- Tip: if a process ever misbehaves this way again, `ls -l /proc/<pid>/cwd`
  shows `… (deleted)` when its working directory was swapped out from under it.

## Tests
