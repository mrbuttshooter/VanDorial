# GenCall v2.2.10 — Fix UAC media port so calls stop dropping at 10s

**The loop UAC now advertises the media port it actually sends RTP from, so strict SBCs accept the audio and hold the call.**

## Symptom

On some routes every answered call was torn down by the carrier at **exactly
10.0 seconds** with `BYE, Q.850 cause=31 "Normal, unspecified"` — even though the
loop was streaming real RTP the whole time. Other routes (whose SBC latches onto
any media source) were unaffected, which masked the bug.

## Root cause

`loop_uac.xml` offered `m=audio [auto_media_port]` — a **per‑call** port SIPp
computes — but `rtp_stream` streams the audio from the single **`-mp`** base
socket. So the SDP advertised one port (e.g. `45530`) while the RTP actually left
from another (e.g. `16388`).

A strict SBC pins media to the port it negotiated. Hearing nothing on that port,
it declared the call media‑dead and cleared it on its ~10s inactivity timer. On
the wire: SDP `m=audio 45530/17760/13756…` vs RTP source `16388` — a permanent
mismatch. The reference "Sigma" box works precisely because its media source ==
its advertised port.

`loop_uac.xml` was the **only** shipped scenario using `[auto_media_port]`; every
other one (`loop_uas`, `basic_call`, `uas_answer`, `stress_test`, …) already uses
`[media_port]`.

## Fix

- `loop_uac.xml`: `m=audio [auto_media_port]` → **`m=audio [media_port]`**, so the
  offered port matches the `-mp` socket `rtp_stream` sends from. The SBC now
  receives audio on the negotiated port and holds the call to its full duration.
- Added a header note explaining why it must stay `[media_port]`, and a
  regression test (`test_loop_uac_advertises_media_port_not_auto`) that fails if
  any `m=audio` offer line drifts back to `[auto_media_port]`.

## Upgrade notes

- No config/API changes. Deploy and **restart the worker** so the new scenario
  is rendered on next campaign launch (the engine copies the template per
  campaign at start).
- Existing campaigns must be **restarted** to pick up the corrected SDP.
- Verify on the wire: the `m=audio` port in your INVITE now equals the RTP source
  port, and carrier `BYE`s at ~10s stop.
  ```bash
  # advertised port:
  sudo tcpdump -i any -n -A 'udp port 5060' 2>/dev/null | grep 'm=audio' | head
  # actual RTP source port (should match):
  sudo tcpdump -i any -n 'udp and udp[8]&0xc0=0x80' 2>/dev/null | head
  ```
